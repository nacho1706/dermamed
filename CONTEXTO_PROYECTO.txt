1. ¿QUÉ ES DERMAMED?
------------------------------------------------------------------------------

DermaMED es un Practice Management System (PMS) desarrollado a medida para un
consultorio dermatológico privado en Argentina. Su propósito central es
digitalizar y centralizar todas las operaciones del consultorio, reemplazando
el uso cotidiano de papel, WhatsApp y planillas de Excel.

El sistema está diseñado para ser la ÚNICA FUENTE DE VERDAD del consultorio:
el lugar donde vive toda la información sobre turnos, pacientes, historia
clínica, facturación e inventario.


2. EL PROBLEMA QUE RESUELVE
------------------------------------------------------------------------------

Antes de DermaMED, el consultorio gestionaba su operativa de forma fragmentada:

  - Los turnos se acordaban por WhatsApp y se anotaban en papel o en una app de
    calendario personal.
  - Los recordatorios a pacientes también se hacían manualmente, lo que generaba
    alta tasa de ausentismo ("no shows").
  - La historia clínica no tenía un repositorio digital centralizado.
  - El stock de productos (cosméticos, insumos médicos) se controlaba con Excel,
    con riesgo constante de inconsistencia.
  - La facturación se hacía de forma independiente sin un registro integrado con
    el resto de la gestión.

DermaMED reemplaza todo esto con un sistema web unificado, accesible desde
cualquier dispositivo con browser.


3. LOS USUARIOS DEL SISTEMA
------------------------------------------------------------------------------

El sistema tiene dos usuarios principales, con roles y necesidades bien
diferenciadas:

  Dra. Roxana Amante — Médica dermatóloga y dueña del consultorio
    - Necesita ver métricas del negocio (facturación, cantidad de turnos, etc.)
    - Necesita acceder y registrar la historia clínica de cada paciente.
    - Necesita control de stock de productos.

  Mariela — Recepcionista del consultorio
    - Es el usuario que más interactúa con el sistema en el día a día.
    - Necesita gestionar turnos de forma ágil.
    - Necesita que el sistema envíe recordatorios automáticos para no tener que
      hacerlo manualmente.

FILOSOFÍA "MARIELA FIRST":

  Cada decisión de diseño del sistema se evalúa bajo la pregunta:
  "¿Esto le ahorra un click a Mariela o le agrega uno?"

  El principio guía todo lo relacionado a la UX: desde los formularios hasta
  los flujos de navegación.


4. FLUJO DE TURNOS: HÍBRIDO-ASISTIDO
------------------------------------------------------------------------------

Un punto clave de diseño es que DermaMED NO busca implementar autogestión de
turnos por parte del paciente. El flujo del MVP es deliberadamente híbrido:

  1. El paciente se comunica con el consultorio a través de WhatsApp.
  2. Mariela acuerda el horario manualmente con el paciente.
  3. Mariela carga el turno en el sistema DermaMED.
  4. El sistema, en forma automática, envía una confirmación del turno y un
     recordatorio 24 horas antes de la cita, vía WhatsApp API.

Este diseño preserva el trato humano y personalizado que caracteriza al
consultorio, mientras automatiza las tareas repetitivas.


5. ARQUITECTURA GENERAL
------------------------------------------------------------------------------

El sistema está compuesto por tres repositorios Git independientes, que
conviven en una carpeta común y se orquestan mediante Docker Compose:

  BACKEND  (carpeta: backend/)
  - API REST construida con Laravel 12 y PHP 8.2+
  - Expone todos los endpoints consumidos por el frontend
  - Maneja autenticación, lógica de negocio, acceso a base de datos

  FRONTEND (carpeta: frontend/)
  - Single Page Application (SPA) / Dashboard construido con Next.js 16
  - Usa el App Router de Next.js con TypeScript estricto
  - Se comunica con el backend exclusivamente vía la API REST

  SETUP    (carpeta: setup/)
  - Contiene la configuración de Docker Compose para todos los servicios
  - Orquesta backend, frontend, PostgreSQL y Adminer

IMPORTANTE: El directorio raíz NO es un repositorio Git. Cada uno de los tres
subdirectorios tiene su propio historial de commits y ramas independientes.


6. STACK TECNOLÓGICO COMPLETO
------------------------------------------------------------------------------

  BACKEND
  ├── Lenguaje:      PHP 8.2+
  ├── Framework:     Laravel 12
  ├── Base de datos: PostgreSQL 17

  ├── Componentes:   Radix UI (componentes accesibles sin estilos propios)
  ├── Iconos:        Lucide React
  ├── Server state:  TanStack Query (React Query) v5

  ├── HTTP Client:   Axios
  ├── Notificaciones: Sonner (toasts)
  └── Fechas:        date-fns v4 con soporte de zonas horarias (date-fns-tz)

  ├── Contenedores:  Docker + Docker Compose

  SERVICIOS (desarrollo local)

  ├── Frontend:      http://localhost:3000
  ├── Adminer:       http://localhost:8081
  └── PostgreSQL:    localhost:5432


7. MODELO DE DATOS
------------------------------------------------------------------------------

Las entidades centrales del sistema son:

  PACIENTES (patients)
    Datos personales y de contacto del paciente. Es el eje de todas las
    demás entidades.

  TURNOS (appointments)
    Cada cita médica. Vincula un paciente, un médico (user con rol doctor)
    y un servicio. Tiene fecha, hora y estado.

  HISTORIA CLÍNICA (medical_records)
    Registro médico por turno. Contiene las observaciones del médico.
    Protegido por rol: solo los usuarios con rol "doctor" pueden ver y
    crear registros clínicos.

  FACTURACIÓN (invoices, invoice_items, invoice_payments)
    Sistema completo de facturación. Cada factura puede tener múltiples
    ítems (servicios o productos) y múltiples pagos (por diferentes
    métodos de pago, ej: efectivo + tarjeta).

  PRODUCTOS Y STOCK (products, stock_movements)
    Catálogo de productos del consultorio con control de inventario.
    Los movimientos de stock registran entradas y salidas.

  SERVICIOS (services)
    Catálogo de prestaciones del consultorio (procedimientos, tratamientos).
    Se usan para vincular a turnos y a ítems de factura.

  USUARIOS Y ROLES (users, roles)
    Usuarios del sistema con asignación de roles en relación muchos a muchos.
    Roles posibles: doctor, receptionist, admin, etc.

  DISPONIBILIDAD MÉDICA (doctor_availability)
    Horarios en los que el médico está disponible para tomar turnos.

  TABLAS DE APOYO (lookup tables)
    payment_methods, voucher_types, health_insurances, brands


8. MÓDULOS DEL SISTEMA (FRONTEND)
------------------------------------------------------------------------------

El dashboard tiene los siguientes módulos:

  /dashboard
    Vista de inicio con métricas y resumen del día.

  /patients
    Listado de pacientes con búsqueda y filtros.
    Ficha individual con datos personales, historial de turnos e historia
    clínica.

  /appointments
    Agenda del consultorio. Gestión de turnos: crear, editar, cancelar.

  /invoices
    Listado y detalle de facturas. Creación de nuevas facturas con selección
    de ítems y métodos de pago.

  /products
    Gestión del inventario: listado, alta de productos, movimientos de stock.

  /services
    Catálogo de servicios del consultorio.

  /users
    Alta y gestión de usuarios del sistema con asignación de roles.


9. SEGURIDAD Y REGLAS DE CUMPLIMIENTO (HEALTHCARE STANDARD)
------------------------------------------------------------------------------

Por tratarse de un sistema que maneja datos médicos sensibles, se aplican
reglas estrictas de seguridad:

  CERO FUGAS DE DATOS
    Nunca se oculta información sensible con CSS (display:none, opacity:0).
    El backend nunca envía datos si el usuario no tiene permiso para verlos.
    El frontend usa renderizado condicional estricto basado en roles.

  SOPORTE DE ROLES MÚLTIPLES
    Un usuario puede tener más de un rol simultáneamente (relación N:M).
    La validación de permisos siempre verifica si el array de roles del
    usuario INCLUYE el rol requerido. Nunca se usa comparación por igualdad
    estricta de un rol único.

  INMUTABILIDAD DE DATOS CLÍNICOS
    Está prohibido el borrado físico (hard delete / DELETE SQL) en las tablas
    críticas: patients, appointments, medical_records, invoices.
    Siempre se usa Soft Delete (borrado lógico): el registro queda en la base
    de datos marcado como eliminado. Esto garantiza integridad referencial
    y cumplimiento del historial médico requerido legalmente.

  PRINCIPIO DE PRIVILEGIO MÍNIMO
    Toda ruta de API y todo componente de frontend está bloqueado por defecto.
    Solo se habilita el acceso si se verifica explícitamente que el usuario
    tiene el rol necesario.

  AUTENTICACIÓN STATELESS CON JWT
    Se usa JSON Web Tokens (JWT) en lugar de sesiones de servidor o cookies.
    Esto es conveniente porque la API es consumida por una SPA y
    potencialmente también por integraciones externas (como un bot de WhatsApp).


10. CONVENCIONES DE DESARROLLO
------------------------------------------------------------------------------

  IDIOMA EN EL CÓDIGO
    Todo el código (nombres de variables, funciones, archivos, rutas URL) se
    escribe en INGLÉS.
    El español se reserva exclusivamente para textos visibles al usuario en
    la interfaz (labels, títulos, mensajes, etc.).

    Incorrecto: /pacientes/nuevo, const obtenerPaciente = () => {}
    Correcto:   /patients/new,   const getPatient = () => {}

  GIT WORKFLOW
    Ramas: main, develop, feature/<nombre>, fix/<nombre>, hotfix/<nombre>
    Los commits siguen el estándar Conventional Commits:
      feat(patients): add search by DNI
      fix(auth): handle expired token gracefully
      chore(deps): update laravel to 12.5

  COMANDOS DOCKER
    Todo comando que requiera PHP, Composer o Node/NPM se ejecuta DENTRO
    del contenedor correspondiente, nunca en el host:

      docker exec backend-dermamed php artisan migrate
      docker exec frontend-dermamed npm run build

      O equivalente desde la carpeta setup/:
      docker compose exec backend php artisan migrate


11. DECISIONES DE DISEÑO DESTACADAS
------------------------------------------------------------------------------

  ¿Por qué JWT y no Laravel Sanctum o sesiones?
    La API está diseñada para ser consumida por múltiples clientes: el
    dashboard Next.js y, en el futuro, un bot de WhatsApp. JWT es stateless
    y no requiere almacenamiento de sesión en el servidor, lo que facilita
    la integración con diferentes clientes.

  ¿Por qué el patrón Factory en el backend?
    Se usa el patrón Factory para separar la lógica de construcción/asignación
    de campos del controller. Esto hace el código reutilizable entre los
    métodos store() y update(), y mantiene los controllers delgados y
    enfocados en la coordinación.

  ¿Por qué Radix UI como base de componentes?
    Radix UI provee componentes primitivos completamente accesibles (WAI-ARIA)
    sin estilos propios, lo que permite construir un design system personalizado
    sobre Tailwind CSS sin tener que luchar contra estilos predeterminados.

  ¿Por qué TanStack Query para el estado del servidor?
    TanStack Query maneja automáticamente caché, revalidación, loading states
    y sincronización de datos del servidor, eliminando la necesidad de gestionar
    manualmente ese estado con Redux o Context.
